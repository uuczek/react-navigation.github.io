Pitch & anti-pitch

在考虑是否使用一个项目来理解开发人员在构建它时做出的权衡时，这篇文章非常有用。 它试图为你解决什么问题，它忽略了哪些问题？ 该项目目前的局限性和人们遇到的常见问题是什么？ 这些是我们认为在为项目做出重要技术决策时应该回答的问题，因此我们尽可能在这里以“优点”的形式记录了这些问题的答案（为什么你应该使用它）和 “缺点”（为什么你不应该使用它）。 如果你认为我们遗漏了什么重要信息，请给我们提 PR ！

优点

  • 一切都用JavaScript编写在React Native原语之上 — 例如，动画使用Animated API及其本机驱动程序选项，以便在主线程上运行动画并产生60 fps的平滑过渡。 使用 JavaScript 编写除了动画和手势等底层基元之外的所有内容都有很多好处：
    • 简单的OTA更新
    • 可调试
    • 可定制
  • 大多数应用的自定义导航，需要使用包装原生导航的API来完成此操作，这需要你编写大量原生代码。
  • 编写自己的导航器很容易，它们可以与标准导航器轻松整合，或者 fork 标准导航器，并根据你想要的外观和 feel 在你的应用中创建你自己的导航器。

缺点

  • API 有时不直观且难以使用，且改进它可能需要破坏性的修改。 我们正在努力使 ”简单的事变得简单，艰难的事变得可能“，这可能需要我们有时更改 API。

    • React Navigation 不直接在 iOS 和 Android 上使用原生导航 API; 相反，它重新创建了这些API的一部分。 这是一个有意识的选择，以便用户能够自定义导航体验的任何部分（因为它是用 JavaScript 实现的），并且能够调试他们遇到的问题而无需学习 Objective C / Swift / Java / Kotlin。
      • 如果您需要指定平台的某些特有功能，最好使用别的包装了该平台 API 的库。 请在 备选方案 中详细了解这些内容，并确保在深度使用之前，了解开发人员所做的权衡！
    • 由于 React Navigation 的大部分逻辑都是用 JavaScript 而不是原生实现的，所以多关注 JavaScript 的线程阻塞是很有用的。
    • 如果需要在iPad上支持主从分解视图，则可能要使用其他库。 尽管可以根据需要自行构建它，但尚未在React Navigation中实现。
